mysql查询慢怎么优化

1.建索引，where order by涉及的列建立索引
2.索引优化，最左匹配原则
3.建立分区，如时间字段
4.利用缓存redis，缓存热数据
5.查询限定数据范围，如查询一个月内的
6.读写分离
7.分库分表，垂直拆分水平拆分
8.数据异构到es
9.冷热数据分离
10.升级OceanBase

1 使用索引：建立合适的索引可以大大提高查询效率，可以使用explain命令查看查询执行计划，确定是否使用索引。

2．减少全表扫描：全表扫描是查询速度慢的主要原因之一。可以通过优化查询语句或者使用索引避免全表扫描。

3.分页查询：如果查询结果集比较大，可以考虑使用分页查询，将查询结果分批返回，避免一次性返回大量数据。

4.优化查询语句：优化查询语句可以使查询更加高效。可以从避免使用子查询、优化where子句条件、避免使用不必要的表连接等方面来优化查询语句。避免使用SELECT *，使用具体的宇段列表；使用JOIN时，使用合适的JOIN方式；使用LIMIT限制查询结果数量等。

5.优化服务器参数：MysQL服务器的参数设置也可以影响查询性能。可以通过修改参数配置文件来优化MySQL服务器。例如，可以通过修改缓存大小、线程池大小等参数来提高MysQL服务器的性能。

6.使用缓存：如果查询的数据不需要每次都进行查询，可以考虑使用缓存来提高查询效率。例如，可以使用Redis缓存查询结果，下次查询时可以直接从缓存中获取结果，而不需要重复查询数据库。

7.使用分区表：将大的表分割成多个分区，可以大大提高查询效率。

微服务是什么

微服务是一种架构风格，它将一个大型复杂的软件系统划分成多个小型的服务，每个服务运行在其独立的进程中，服务之间通过轻量级的通信机制相互协作，可以独立地部署、扩展和维护。

微服务架构通常由以下组件组成：
1 服务：微服务架构中的基本单位，每个服务都是一个独立的、自治的、精简的业务功能。一个服务可以由多个进程、容器或虚拟机组成，每个服务都有自己的数据存储和处理能力。
2．服务注册与发现：微服务架构中需要一种机制来管理所有的服务，将其注册到服务注册中心并提供发现服务。服务注册中心可以帮助服务实例注册、发现、注销和维护的过程。
3. 负载均衡：在微服务架构中，服务实例的数量可能非常多，需要一种机制来分配请求流量。负载均衡器可以帮助将请求流量分配到不同的服务实例上，以实现高可用、高性能和高弹性。
4. API网关：微服务架构中的AP!网关可以作为服务访问的入口，为客户端提供一个整体的视图。API网关可以提供安全性、监控、缓存、路由等功能。

微服务架构的优点包括：
1模块化：微服务架构将系统拆分为多个小型服务，每个服务都可以独立开发、测试和部署。
2．可扩展性：微服务架构可以水平扩展，通过增加或减少服务实例的数量，可以灵活地调整系统的容量和性能。
3. 可靠性：由于微服务架构中的服务是自治的，如果某个服务出现故障，其他服务不会受到影响。
4. 技木异构性：微服务架构可以使用不同的编程语言、框架和技术栈，可以采用最适合服务的技术栈。
5. 可维护性：微服务架构可以让开发人员专注于单个服务的开发和维护，降低了系统的复杂度和维护成本。

综上所述，微服务架构是一种分布式系统的设计方法，它的目标是将大型系统拆分为多个小型服务，以提高系统的可维护性、可扩展性、可靠性和灵活性。

微服务之间是如何通讯的

微服务之间可以通过多种方式进行通讯，常见的有以下几种：

  1. HTTP/REST： 采用HTTP协议进行通讯，每个服务提供RESTful API，可以通过HTTP请求调用其他服务的API接口。
    HTTP/REST通讯简单，易于理解和调试，但有时候会存在性能瓶颈。

  2. gRPC：采用Google开源的gRPC框架进行通讯，基于HTTP 2.0协议，数据传输效率高，同时支持多种编程语言和多种序列化协议，能够支持更加复杂的场景。

  3. 消息队列：采用消息队列进行异步通讯，每个服务可以发布和订阅消息，服务之间通过消息传递来进行通讯。消息队列可以解耜服务之问的直接依赖关系，提高系统的可靠性和伸缩性，但也可能会引入一些复杂性。(发送消息监听消息)

  4. RPC： 远程过程调用 (Remote Procedure Call， RPC)是一种传输协议和通信协议，用于通过网络从远程计算机程序请求服务。每个服务都暴露出一些远程调用接口，其他服务可以通过RPC调用这些接口。（一个节点请求另一个节点提供的服务，优点是简单无中间件代理 缺点是只支持请求响应模式，不支持通知订阅）

不同的通讯方式各有优缺点，具体选择需要根据实际业务场景和需求来确定。
例如，如果需要快速迭代开发，HTTP/REST可能是更好的选择；如果需要高效的数据传输和支持多种编程语言，gRPC是更好的选择；如果需要异步通讯和解耩服务之间的依赖关系，消息队列可能是更好的选择。

在微服务架构中，服务之问的通讯是通过网络进行的，因此需要考虑网络延迟、故障率、安全性等因素，需要采用相应的技术手段来保证通讯的可靠性和安全性。

RPC和HTTP的区别

RPC：就像调用本地的函数一样去调用远程函数。
RPC是一个完整的远程调用方案，通常包括通信协议和序列化协议。
通信协议包括：http协议（grpc用的是http2）、自定义报文的tcp协议（dubbo）。
序列化协议包括：基于文本编码的xml、json，基于二进制编码的protobuf、hessian等。

HTTP，超文本传输协议，是从万维网服务器传输超文本到本地浏览器的传输协议，是通信协议，不是完整的远程调用框架。

为什么RPC有的用自定义TCP协议进行远程通信，而不是HTTP

HTTP好处是支持连接池复用，建立一定数量的连接不断开，不会频繁地创建和销毁连接。
但是http传输协议的header部分有很多冗余部分，例如content-type、last-modified、expires等，即使http body用二进制编码，header头的键值依然文本编码，占用字节数。
自定义的TCP协议可以精简传输内容，传输速度更高，可以自定义tcp协议的报文。

因此，对于性能要求比较高的系统来说，通常使用自定义tcp协议的rpc来做后段进程通信。

其实，使用http协议比较多的还是前后端的通信，原因在于主流网页浏览器都支持http协议，而且http在缓存、幂等重试乃至cookie这种浏览器安全相关的方面做了很多功夫。

grpc和rpc的关系

gRPC是一种基于Google开源的RPC (Remote Procedure Call，远程过程调用）框架实现的通用 RPC 框架。gRPC 支特多种编程语言，并且基于HTTP/2 协议实现，提供了诸多的优化和特性。

RPC，它允许客户端应用程序像调用本地服务一样调用远程服务。在RPC 中，客户端应用程序可以调用远程服务，而无需了解服务实现的细节，而服务实现者也不需要了解调用方的具体实现。

gRPC 是在RPC 的基础上，使用Google 开源的Protocol Buffers 作为 IDL (Imntertace Definition Lanuege接口定义语言）和数据序列化机制，提供了高效的数据传输和多语言支持等功能。gRPC 基于 HTTP/2 协议实现，不仅可以提供更高效的数据传输，还可以支持双向流、多路复用等诸多特性，适用于分布式系统中的服务间通信。

综上所述，gRPC 是在 RPC 的基础上，使用 Protocol Buffers 作为 IDL 和数据序列化机制的一种通用 RPC 框架。gRPC基于 HTTP/2 协议实现，提供了高效的数据传输和多语言支持等特性，适用于分布式系统中的服务问通信。

rpc是完整的远程调用方案，http是通信协议，rpc可以用http通信协议也可以用udp，grpc是一种rpc，它使用的http2.0.这样说有问题吗？？？

RPC (Remote Procedure Call)，远程过程调用和 HTTP (Hypertext Transfer Protocol，超文本传输协议）都是用于网络通讯的协议，但它们的底层实现机制和应用场景有一些不同。

RPC 协议可以支持多种底层的通讯机制，如TCP，UDP，RMI 等，它的通讯方式更加灵活和高效。与 HTTP 协议相比，RPC 协议的数据传输效率更高，因为它不需要像 HTTP 协议那样在每次请求和响应时都需要传输大量的头部信息。

gRPC 是一种基于 Google 开源的 RPC 框架实现的通用 RPC框架，它是基于 HTTP/2 协议实现的，因此在底层数据传输时使用了 HTTP/2 协议。HTTP/2 协议相比于 HTTP/1.× 协议，具有更高的性能和更好的效率，例如多路复用、二进制传输格式等特性，这些特性可以帮助gRPC 提高性能和效率。

综上所述，RPC 和HTTP 都是用于网络通讯的协议，但它们的底层实现机制和应用场景有所不同。虽然 gRPC 底层使用了HTTP/2 协议，但它并不是 HTTP 协议的直接实现，而是在 RPC 协议的基础上发展而来的通用 RPC 框架。

HTTP有什么特点

1.HTTP允许传输任意类型的数据，传输的类型由content—type加以标记。

2.无状态，对于客户端每次发送的请求，服务端都认为是一个新的请求，上一次会话和下一次会话之间没有联系。

3.支持客户端/服务端模式，web服务端负责处理请求和响应，客户端负责发送请求和接收响应。

HTTP长连接

复用TCP连接，多个HTTP请求复用同一个TCP连接，这就节省了TCP连接建立和断开的消耗。

HTTP1.1和HTTP2.0的区别

HTTP1.0 : 默认使用短连接，浏览器和服务器每进行一次HTTP操作，就建立一次连接，任务结束就中断连接。

HTTP1.1起 : 默认使用长连接，要使用长连接，客户端和服务端的HTTP首部的connection都要设置为keep-alive，才能支持长连接。

HTTP1.1和HTTP2.0的区别 :

  1. HTTP2.0支持新的二进制格式：HTTP1.1基于文本格式传输数据，HTTP2.0采用二进制格式传输数据，解析更高效。
  2. HTTP2.0支持多路复用：在一个连接中，允许同时发送多个请求或响应，并且这些请求或响应能够并行的传输而不被阻塞。
  3. HTTP2.0支持头部压缩：HTTP1.1的header带有大量信息，而且每次都要重复发送，HTTP2.0把header从数据中分离，并封装成头帧和数据帧，使用特定算法压缩头帧，有效减少头信息大小。并且HTTP2.0在客户端和服务端记录了之前发送的键值对，相同的数据不会重复发送，减少了冗余数据，降低开销。
  4. HTTP2.0支持服务端推送：HTTP2.0允许服务端向客户端推送资源，无需客户端发送请求到服务端获取。

HTTP通信安全吗

HTTP不安全，因为是明文传输。
可以使用HTTPS：在HTTP基础上加入SSL/TSL协议，依靠SSl证书来验证服务器的身份，为客户端和服务端之间建立SSL通道，确保数据传输安全。

RESTFUL是什么

REST：resource representational state transform 资源表现层状态转移。

RESTFUL：满足REST（对资源的访问状态的变化通过URL的变化（HTTP的method）表现出来），是一种网络应用的设计风格。

REST描述的是网络中client和server的一种交互形式，通过URL就知道要什么资源，通过HTTP method就知道要干什么，通过HTTP status code就知道结果如何。

GET：获取资源；POST：添加资源；PUT：修改资源；DELETE：删除；
URL原则上不用动词。
用URL定位资源，用HTTP描述操作。

REST优点：
1.风格统一
2.面向资源，一目了然，具有自解释性
3.充分利用HTTP协议本身语义

正向代理和反向代理

正向代理：代理的对象是客户端，隐藏了真实的请求客户端，服务端不知道真实的客户端是谁，客户端请求的服务都被代理服务器代替去请求。例如：科学上网工具。

反向代理：代理的对象是服务端，隐藏了真实的服务端，nginx反向代理服务器，可以用来做负载均衡。

网络分层：

应用层：为应用程序提供交互服务DNS、HTTP、SMTP
传输层：TCP、UDP
网络层：选择合适的路由，交换节点 IP协议
数据链路层：传送帧
物理层：比特流的透明传输

GET和POST区别

1.最本质的区别在规范上，GET请求用来获取资源，也就是进行查询操作的，POST请求用来传输实体对象的，进行添加，修改和删除。

2.GET请求参数通过url传递，POST请求参数放在请求体中。

3.GET请求，浏览器会把请求头和请求体一并发送出去，而POST请求浏览器会先发送请求头，服务器响应100 continue，浏览器再发送请求体。

@Autowired和@Resource的区别

@Autowired是spring的注解，如果按照名称匹配接口不同实现类需要结合@Qualifier，默认根据类型注入。

@Resource是j2ee的注解，默认按byName自动注入，name和type属性都不指定，spring容器将通过反射默认按byName注入。

java反射是什么

动态获取信息，动态调用对象方法 - java反射。

在运行过程中，任意一个类，知道这个类的属性、方法；任意一个对象，可以调用它的方法/属性。

反射应用场景：
1.JDBC连接数据库时使用Class.forName()，通过反射加载数据库的驱动程序。
2.IDEA利用反射动态解析对象类型、结果，提示对象属性、方法。
3.Web服务器中利用反射调用Servlet的service方法。
4.JDK动态代理底层依赖反射实现。

String、StringBuilder、StringBuffer

1.可变性：String不可变
2.线程安全：String线程安全，StringBuilder非线程安全，StringBuffer线程安全，内部使用synchronized同步。
